## The Day-30 gate (PRE-COMMITTED — written before launch, not after)

**CONTINUE** toward SERV-01 paid pilots if ANY of:

1. **≥150 real weekly npm downloads** (measurement method below), sustained
   in the final week (Jul 29–Aug 04), or
2. **≥50 GitHub stars** on nerviq/nerviq (from 6 at sprint start), or
3. **≥1 genuine inbound lead**: a human we did not contact first, identifiable
   (name/org), asking about usage/pilot/pricing — via the site forms
   (`npm run leads:list`), email, or a substantive GitHub issue/discussion
   from a real account with history.

**Else SUNSET, gracefully, that week — no third option:**
- Archive repos with an honest README note (the audit's §5 text).
- Unpublish nothing; npm package stays installable.
- Extract stale-reference detection as standalone **`npx agent-doc-lint`**
  (the `nerviq drift` entrypoint built in Wave 1 is the seed — near-zero
  maintenance, single wedge, MIT).
- Keep the CC0 state-of-agent-governance dataset as a yearly artifact.

**Anti-gaming clauses** (decided now so Day-30-us can't lawyer them):
- Downloads driven by our own CI, install-tests, or republish blitzes don't
  count (that's what created the April 4,504/wk artifact). During the sprint:
  no scheduled CI jobs that `npm install @nerviq/cli`, and version publishes
  are capped at 2 (v1.31 + one fix release) to keep mirror noise flat.
- Stars from personal-network asks count only up to 10; the target is 50 so
  the network can't fake the signal.
- A "lead" that is actually a vendor selling us something is not a lead.

### Measuring "real" weekly downloads (excluding bot/mirror noise)

Noise floor first: the audit measured **438 downloads/month ≈ ~100/week** with
zero real users — that is this package's ambient registry noise (mirrors,
security scanners, replicators). Method:

1. **Baseline**: record the 4 pre-launch weeks (Jun 8 – Jul 5) from
   `https://api.npmjs.org/downloads/range/2026-06-08:2026-07-05/@nerviq/cli`
   → weekly noise floor N (expected ~100–110/wk). Commit to
   `docs/sprint-metrics/baseline.json` on Day 7.
2. **Signal**: final-week downloads W (Jul 29–Aug 04 from the same API).
   **Real downloads = W − N.** Gate #1 passes iff `W − N ≥ 150`.
3. **Cross-checks** (any one is confirming, not required): npm per-version
   split (real users install `latest`; mirrors re-fetch every version
   uniformly — a spike concentrated on v1.31 is real); publish-day exclusion
   (ignore the 48h after any publish — mirror refetch spike); correlation
   with an independent channel (GitHub traffic/clones, site visits) — real
   downloads co-move with referrers, noise doesn't.
4. All numbers pulled on Day 30 by script, pasted into the decision doc
   verbatim. No adjustments after seeing them.
